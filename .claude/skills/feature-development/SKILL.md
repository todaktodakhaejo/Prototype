# 기능 개발 워크플로우 (main 직접 작업)

이 저장소는 **빠른 프로토타이핑** 단계라, 피처 브랜치/PR 없이 **main에서 직접 작업하고
바로 push**한다. (2026-05-31, 사용자가 "A안" 선택 — 브랜치가 쌓여 작업이 사라진 것처럼
보이는 문제를 없애기 위함.)

## 절차

### 1. 시작 전 — main 최신화
```bash
git checkout main
git pull origin main
```
로컬에 미커밋 변경이 있으면 먼저 확인/정리한다.

### 2. 작업 — 작은 단위로 구현
- 요청 기능을 main에서 직접 구현한다. 한 번에 너무 많이 바꾸지 않고 작은 단위를 선호한다.
- Node/npm은 에이전트 셸 PATH에 안 잡히므로 빌드/실행 검증은 사용자에게 맡긴다([[node-not-installed]]).

### 3. 완료 후 — 커밋 → main에 push
**작업이 전부 끝나면** 매번 묻지 않고 진행한다(이 절차 자체가 상시 승인이다).
```bash
git add -A
git commit -m "<요약>

<본문: 무엇을 왜 바꿨는지>

Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>"
git push origin main
```
- push 후 커밋 해시와 결과를 사용자에게 보고한다.
- 변경 후 진행 로그는 `CLAUDE.md`에 한 줄 남긴다(인수인계용).

## 주의
- **피처 브랜치·PR을 만들지 않는다.** 모든 작업은 main 직선 히스토리로 쌓는다.
  (이전에 브랜치를 쌓다가 미머지 작업이 "사라진 것처럼" 보이는 사고가 있었음 → main 단일화로 해결.)
- push는 외부로 나가는 행위이므로 "작업 완료" 시점에만 수행한다. 미완성 상태로 push하지 않는다.
- 되돌리기 쉽도록 **작은 단위로 자주 커밋**한다. 한 커밋에 여러 기능을 뭉치지 않는다.
- 만약 위험한 대규모 변경이거나 사용자가 리뷰를 원하면, 그때만 예외적으로 브랜치를 쓰고
  사용자에게 확인한다.
